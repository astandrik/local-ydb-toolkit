Greptile Summary
  This PR extends the backup/restore toolkit with path-scoped dump and restore, a new read-only local_ydb_list_dumps MCP tool, and optional post-restore verification hooks (scheme describe + bounded count queries). Input validation covers path traversal and injection for dump/restore paths, with shell-quoting fixes applied to rollback and verification strings.

  - New local_ydb_list_dumps tool enumerates host-side dump directories that contain a tenant subdirectory and exposes them as structured output for use in subsequent restore calls.
  - Path-scoped dump/restore lets callers target a tenant-relative subdirectory instead of always operating on the full tenant root; normalizeDumpName and normalizeYdbRelativePath guard against traversal attacks.
  - Post-restore verification hooks allow optional scheme describe and SELECT COUNT(...) checks to run as additional specs in the same mutating operation; the count-query validator guards against multiple statements and UNION but does not restrict subqueries after FROM.

Confidence: 4/5
  The change is a well-structured feature addition with good path-traversal guards and shell-quoting fixes; the issues found are non-blocking quality concerns.

  Path normalization and shell quoting are handled carefully throughout. The count-query validator blocks the most dangerous multi-statement patterns but leaves subqueries after FROM unconstrained, so verification hooks can silently query tables other than the restored path. The dumpName Zod validator accepts empty strings, causing a silent fallback instead of a clear error. The CI integration test leaves YDB schema objects behind, which would break reruns on persistent containers. None of these block a careful merge, but they are real gaps in the new validation surface.

  packages/core/src/operations/tenant.ts (normalizeCountQuery logic) and scripts/ci/verify-live-mcp-server.mjs (missing YDB schema cleanup) warrant the closest look before merging.

1. P2 comment packages/core/src/operations/tenant.ts:217-219
  Misleading validation error for nested-paren COUNT expressions

  The [^)]{1,256} quantifier stops at the first ) inside the COUNT argument. A perfectly valid query like SELECT COUNT(NULLIF(x, 0)) FROM table fails because the inner ) terminates the match before the outer ), so \)\s+from\b never sees the FROM. The error thrown is "must start with SELECT COUNT(...) and include FROM" — which describes neither the actual problem nor the unsupported construct, making the constraint invisible to callers.

2. P2 comment packages/core/src/operations/tenant.ts:208-221
  Subquery bypass in count-query validation

  The regex anchors the start of the query (^SELECT COUNT(...)...FROM) but leaves everything after FROM unconstrained. A query like SELECT COUNT(*) FROM (SELECT id FROM admin_table WHERE secret = 'x') AS t passes every check — no ;, no UNION, starts with SELECT COUNT — yet targets an arbitrary table instead of the restored path. Since the spec is passed to ydbCli and executed inside the container, this allows any reader of the MCP interface to run unrestricted reads against any tenant table as a "verification" hook.

4. P2 comment scripts/ci/verify-live-mcp-server.mjs:296-320
  YDB schema objects not cleaned up after the backup/restore CI test

  verifyBackupRestore creates ci_backup_src/items and then restores into ci_backup_dst/items but never issues DROP TABLE or scheme rmdir for either path. The finally block only removes the host-side dump directory. On a reused container (e.g., a developer running the integration test locally on a persistent environment), the subsequent CREATE TABLE \${sourceTable}\ … call without IF NOT EXISTS will fail with a "table already exists" error, aborting the test before the restore cycle is verified. Compare with verifySchemaApply, which explicitly drops its tables in cleanup.
