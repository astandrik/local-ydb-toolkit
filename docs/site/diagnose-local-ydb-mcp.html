Diagnostics

Start broad, read-only, then route by the YDB health signal.

@astandrik/local-ydb-mcp gives AI agents a safe diagnostic path for Docker-based local-ydb stacks. The preferred flow is status report, YDB healthcheck, then targeted checks based on issue type.

Recommended diagnostic order

  1. Run local_ydb_check_prerequisites on a new host or profile.
  2. Run local_ydb_status_report to capture Docker, tenant, node, auth, and health context.
  3. Run local_ydb_healthcheck when a fresh YDB self-check is needed.
  4. Route by issue type instead of repairing automatically.

Route by symptom

Symptom or issue type Next read-only tools
Storage issues local_ydb_storage_placement, then local_ydb_container_logs
Compute, compute pool, tablet, or node issues local_ydb_nodes_check, local_ydb_tenant_check, and dynamic container logs
Database or scheme symptoms local_ydb_database_status and local_ydb_scheme
Auth symptoms local_ydb_auth_check

Agent prompt

The MCP server includes a local_ydb_diagnose_stack prompt. It instructs the client to inspect current state, summarize observed evidence, and avoid automatic repair.

Mutating repair tools should be called first without confirm. Execute with confirm: true only after the exact plan is approved.
